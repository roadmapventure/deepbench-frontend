## 2. CONTEXT

DB Assembly (`api/prompt/db-assembly.js`) builds the Prompt Request server-side for every agent call. Its Identity section currently reads from `agent_configs` (role_prompt rows) with OR-fallback to skill profile objective/method. It has no access to the agent's name, role, or specialty — that data lives in `agents.js`, which is client-only and unavailable server-side.

**AA-58** creates a new `agents` Supabase table as the server-side professional card for all agents. Required because DB Assembly needs agent identity data at prompt assembly time.

**AA-66 (partial)** updates the Identity section in DB Assembly from OR logic to additive assembly: all non-blank sources combined — agents table (name + role + specialty), all role_prompt entries from agent_configs (not just is_default), and skill profile objective + method.

**agent_id key decision (Path B):** The agents table PK uses the name slug (`'michelle'`, `'dan'`) to match the existing system. A `code` column (PP-01, PS-01, etc.) is included for display and future migration. The full ID migration to code-as-canonical-identifier is deferred to S-BENCH-FULL-MIGRATE.

**What is NOT in scope:**
- Migrating call sites from `agent.id` to `agent.code` (future session)
- Seeding editor agents AG-14/15/16 (names TBD in S-EDITOR-01 — skip for now)
- Any UI changes
- Full agents.js migration (salary, stats, flags — S-BENCH-FULL-MIGRATE)
- agent_configs table changes

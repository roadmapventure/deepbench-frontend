## 2. CONTEXT

### Root cause

`AIActivityPanel.jsx` has a local `AGENT_NAMES` lookup table (lines 9–17) used to render the By Agent section. It maps agent id → display name + code. When an agentId is not found in this table, the fallback is `{ name: d.agentId, code: "—" }` — meaning the raw id renders as the name with no code.

`michelle` and `susan` were never added to `AGENT_NAMES` when those agents were added to the roster. This caused two QA failures in S-AI-ATTR-01:

1. **Susan shows as "susan · —"** — raw id fallback, no full name, no code
2. **Michelle not visible at all** — Michelle's planning calls do write agentId to Supabase correctly (fix from S-AI-ATTR-01 shipped), but the panel hydrates from Supabase on open. Older DB rows have null agent_id. New rows accumulate with `agent_id:"michelle"` after the S-AI-ATTR-01 deploy. Once Michelle appears, she will render with full name + code after this patch.

### Fix

Add `michelle` and `susan` to `AGENT_NAMES`. One location, two lines. No other changes.

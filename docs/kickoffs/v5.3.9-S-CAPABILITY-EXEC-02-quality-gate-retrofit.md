## 2. CONTEXT

`quality-gate.js` (Owen Marsh, the Proofreader) is the second of two capabilities `ARCHITECTURE.md` §19b names for retirement onto `api/capabilities/execute.js` — `channel-intelligence.js` was the first, done in `S-CAPABILITY-EXEC-01`. Like `channel-intelligence.js` before it, `quality-gate.js` hand-rolls its own Anthropic call and `ai_activity_log` write (`reviewOnce()`/`logActivity()`, lines 51–126) instead of using the already-generic `assemblePrompt()` → `enrichPrompt()` → `sendRequest()` pipeline. Retiring it onto `execute.js` needs zero changes to `execute.js` itself — this session is Supabase data (Task 1) + a shared-pipeline bug fix that happens to be surfaced by this retrofit (Tasks 2–3) + the deletion (Task 4).

**Two supporting facts, verified live against Supabase, not assumed:**
- `qg-review-intent` already carries the correct `llm_model`/`max_tokens` (`claude-haiku-4-5-20251001` / 600) — matches what `quality-gate.js`'s own `callAnthropic()` hardcodes today. `traits` is `{}` (empty) — Task 1 backfills it with the exact schema currently hardcoded as `QUALITY_GATE_TOOL.input_schema`.
- Live query of `ai_activity_log` confirms the `AI-41` bug's real impact: 37 rows already carry the generic `ai_type: 'request-receivable'` literal, including every `plan.js` call ever routed through `sendRequest()` — this is a platform-wide gap, not something introduced by or limited to this retrofit.

**Root cause of `AI-41` (already logged in `docs/FEATURES.md`, restated here only because Task 2 depends on the exact mechanism):** `request-receivable.js`'s Step 4 log write (line 263) hardcodes `ai_type: 'request-receivable'` regardless of which capability called it. `capability_slug` is already an existing parameter on `sendRequest()` (line 54) — Task 2 is a one-line change to use it.

**`AI-40` resolution note (session-specific finding, not a standing rule — write this into `docs/FEATURES.md` at close-out):** `AI-40` was filed as `AI_TYPE_TO_SERVICE`'s `request_receivable` key (underscore) never matching the actual logged `ai_type` value `request-receivable` (hyphen). Investigation this design session found the "By Service" aggregation (`useAIActivity.js` line 280) already falls back to `AI_TYPE_TO_SERVICE[ai_type] || ai_type` — meaning an unmapped `ai_type` that already equals a `SERVICE_CATALOG` slug (which `'request-receivable'` does, line 20) resolves correctly today regardless of the typo. `AI-40`'s literal typo is dead code, not a live bug — it's superseded by Task 2's fix rather than patched directly, since after Task 2 the Step 4 log no longer writes the literal `'request-receivable'` value at all (except as the `|| 'request-receivable'` fallback if `capability_slug` is ever missing, which `sendRequest()`'s existing guards make unreachable in practice). No code change needed for `AI-40` specifically; mark it resolved-as-superseded, not fixed, in `FEATURES.md`.

const SUPABASE_URL = process.env.SUPABASE_URL;
const SUPABASE_KEY = process.env.SUPABASE_SERVICE_KEY;

let pass = 0, fail = 0;
function check(label, cond) {

async function supaHeaders() {

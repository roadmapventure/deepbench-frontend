## 1. CONTEXT

The AI Audit panel currently groups AI activity by informal "AI type" strings (`planning`, `routing`, `rag_briefing`, etc.) that predate the AI Services model. These strings are a mashup — some map to full services, some map to sub-steps of a service (e.g. `similarity` is just the embeddings step of Knowledge Retrieval, `summarization` is just the synthesis step of Knowledge Reinforcement).

This session rebuilds the panel on the AI Services model:

**Five sections (replace current "Phase 1 AI Types" + "Future Tracking"):**
1. **By Service** — 14 named DeepBench AI Services, one row each. Type badge (AI/Hybrid/Logic) + service name + patterns inline + calls/cost/latency. No expandable detail.
2. **By Pattern** — 10 industry AI Patterns, one row each. Name + description + calls/cost. Inactive patterns greyed. This is a catalog view showing which industry patterns DeepBench implements.
3. **By LLM** — unchanged from current.
4. **By Agent** — unchanged from current.
5. **Roadmap** — collapsed by default. Services Now/Next/Later + Patterns Now/Next/Later. Replaces the old "Future Tracking" phase-2 section.

**New MCP Roadmap tab** — second tab (between Activity Log and Architect Checklist). Static content: 7 MCP exposure surfaces with tier badge, one-sentence description, and caller type. Bidirectional callout on MC-06/MC-07.

**Header strip** — grows from 4 to 5 stats: Total Calls · Total Cost · Services Active · Patterns Active · Models in Use.

**Data continuity** — no data is lost. Old `ai_type` strings in Supabase are remapped to `service_slug` values at read time via a static `AI_TYPE_TO_SERVICE` table in `useAIActivity.js`. The DB is not touched in this session. The `ai_services` and `ai_patterns` Supabase tables (AI-25, AI-26) will be created in S-INFRA-01 — this session uses client-side catalogs as the source of truth.

**api/ count:** 12 files before this session. This session adds no new `api/` files. Count stays at 12.

```js
  // FEATURE: AI-23 — Aggregate by Service (remaps old ai_type to service_slug)
  const byService = {};
  for (const svc of SERVICE_CATALOG) {
    const entries = log.filter(e => (AI_TYPE_TO_SERVICE[e.type] || e.type) === svc.slug);
    const cost    = entries.reduce((s,e) => s + (e.cost||0), 0);
    const latencies = entries.filter(e=>e.latencyMs).map(e=>e.latencyMs);
    byService[svc.slug] = {
      ...svc,
      total:      entries.length,
      cost,
      avgLatency: latencies.length ? Math.round(latencies.reduce((a,b)=>a+b,0)/latencies.length) : null,
    };
  }

  // FEATURE: AI-23 — Aggregate by Pattern (roll up from services that declare the pattern)
  const byPattern = {};
  for (const pat of PATTERN_CATALOG) {
    const svcsUsingPattern = new Set(SERVICE_CATALOG.filter(s => s.patterns.includes(pat.name)).map(s => s.slug));
    const entries = log.filter(e => svcsUsingPattern.has(AI_TYPE_TO_SERVICE[e.type] || e.type));
    const cost    = entries.reduce((s,e) => s + (e.cost||0), 0);
    byPattern[pat.slug] = { ...pat, total: entries.length, cost };
  }

  const servicesActive     = Object.values(byService).filter(s => s.total > 0).length;
  const patternsActiveCount = PATTERN_CATALOG.filter(p => p.active).length; // static: 8 of 10
```

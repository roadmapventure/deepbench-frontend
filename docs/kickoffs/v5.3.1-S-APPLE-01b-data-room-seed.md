**Corrected during this design session's Architect Review (2026-07-01) — read before starting, this changes the migration from what earlier docs described:**
- `status` already exists live on `knowledge_entries` (default `'active'`, all 34 current rows use it) — it is **not** a new column, it just needs `'superseded'`/`'archived'` as additional valid values going forward.

- The existing free-text `category` column (already live, already used by other agents — Brent "Portal Navigation", Michelle "Capability Registry" etc.) is reused as the `business_category` tag (AA-70) — no new column for that.

```js
// scripts/seed-apple-data-room.mjs — S-APPLE-01b one-off seed, delete after running
import fs from "fs";

const SOURCE_PATH = "docs/APPLE-DATA-ROOM-SOURCE-DATA.md";
const INGEST_URL = process.env.INGEST_URL; // dev preview URL + /api/ingest
const BYPASS_HEADER = process.env.VERCEL_BYPASS; // deepbenchDeploymentProtectionByp, per reference-vercel-bypass

const MANIFEST = [ /* ... paste full manifest from above ... */ ];

const raw = fs.readFileSync(SOURCE_PATH, "utf-8");

function extract(startHeading, endHeading) {
  const startIdx = raw.indexOf(startHeading);
  if (startIdx === -1) throw new Error(`startHeading not found: ${startHeading}`);
  const endIdx = raw.indexOf(endHeading, startIdx + startHeading.length);
  if (endIdx === -1) throw new Error(`endHeading not found: ${endHeading}`);
  let block = raw.slice(startIdx, endIdx).trim();
  block = block.replace(/\n---\s*$/, "").trim(); // strip trailing separator if captured
  return block;
}

let successCount = 0;
for (const entry of MANIFEST) {
  const content = extract(entry.startHeading, entry.endHeading);
  const res = await fetch(INGEST_URL, {
    method: "POST",
    headers: { "Content-Type": "application/json", "x-vercel-protection-bypass": BYPASS_HEADER },
    body: JSON.stringify({
      title: entry.title,
      category: entry.category,
      data_type: entry.data_type,
      citeable: entry.citeable,
      geo: entry.geo || null,          // not a real column — folded into content/teaching_note if needed, see note below
      program_area: entry.program_area || null,
      partner_id: entry.partner_id || null,
      period: entry.period || null,
      content,
      is_baseline: true,
      agent_id: "apple-cso-data-room",
      tenant_id: "global",
      jurisdiction: "All",
      priority: 50,
      triggers: [],
    }),
  });
  if (!res.ok) {
    console.error(`FAILED: ${entry.title} — ${await res.text()}`);
    continue;
  }
  successCount++;
  console.log(`OK: ${entry.title}`);
}
console.log(`\n${successCount}/${MANIFEST.length} rows seeded.`);
```

**Important gap to resolve before running this script — flagging, not deciding:** `geo`, `program_area`, `partner_id`, and `period` are **not real columns** on `knowledge_entries` (confirmed against live schema, Task 1's ALTER TABLE does not add them — they were never part of the §7 versioning migration). The manifest above and the design conversation both assumed they'd be stored as queryable fields, but there's no column for them. Three options, pick one before running the script:
1. **Add 4 more columns** (`geo text`, `program_area text`, `partner_id text`, `period text`) to Task 1's migration — cheapest fix, keeps them queryable, but wasn't scoped/approved in the design session.
2. **Fold into `teaching_note`** — `ingest.js` already has this free-text field for "admin instruction to agent on how to use this doc." Store as a small prefixed block, e.g. `teaching_note: "geo: Southeast Asia | program_area: NPI | partner_id: signal-mobile-th-id | period: 2025-Q4"`. No schema change, but not structured/queryable — AA-71's future trend-comparison feature would need to parse this back out of a text field, which is fragile.
3. **Fold into `content`** — least clean, mixes metadata into the RAG-embedded prose itself.

None of these were decided in the design conversation — this surfaced while writing the seed script. **Do not guess — confirm with John before running Task 3.** Recommendation: option 1 (add the 4 columns) is the cleanest and keeps AA-71 buildable later without a text-parsing workaround, but it's John's call since it wasn't part of the approved §7 migration scope.

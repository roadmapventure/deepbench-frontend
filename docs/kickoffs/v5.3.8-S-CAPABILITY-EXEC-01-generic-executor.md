## 2. CONTEXT

`channel-intelligence.js` and `quality-gate.js` each hand-roll their own Anthropic API call and `ai_activity_log` write, duplicating logic that already exists generically in `request-receivable.js`'s `sendRequest()` — proven working today by `plan.js`. Root cause on record in the `S-APPLE-02b` kickoff doc: a past session was explicitly told to read `request-receivable.js` "as precedent only, not imported or modified" and copy its pattern into a new file, rather than import and extend it. That instruction is now a named, banned anti-pattern (`ARCHITECTURE.md` §19b).

This session builds the fix: **`api/capabilities/execute.js`**, one generic route that runs `assemblePrompt()` → `enrichPrompt()` → `sendRequest()` for *any* capability, driven entirely by `{capability_slug, intent_slug, agent_id, task_context}`. It contains zero capability-specific code — no `if (capability_slug === 'x')`, ever. Going forward, a new capability is Supabase rows, not a new file.

Two supporting facts, verified live against Supabase, not assumed:
- `ci-routing-intent` (`llm_model: claude-haiku-4-5-20251001, max_tokens: 300`) and `ci-answer-intent` (`llm_model: claude-sonnet-4-6, max_tokens: 1500`) **already carry the correct values** — `db-assembly.js` just never reads them off an Intent-type Skill Profile, only a Format-type one. Task 2 fixes this by mirroring the existing `format` branch's logic (`db-assembly.js` ~line 114) onto the `intent` branch.
- `traits` is `{}` (empty) on both Intent Skill Profiles today — there is no schema data yet. Task 1 backfills it with the exact JSON schemas currently hardcoded as `ROUTING_TOOL`/`ANSWER_TOOL` in `channel-intelligence.js`.

**Two disclosed, accepted behavior changes from retiring `channel-intelligence.js` (not "byte-identical" — read this before building):**
1. **Response shape changes.** `channel-intelligence.js` today returns `{ result: {...}, debug: {...} }`. Because `execute.js` routes through `sendRequest()`, the new canonical shape is `{ deliverable_id, title, handler, guardrails_passed, violations, patterns_used, debug: {...}, content: {...} }` — the structured output (intent/confidence/etc., or answer/citations/etc.) is now at `.content`, not `.result`. **There is no current frontend caller** (grepped `src/` — zero matches; the Market Intelligence screen this feeds, `S-MARKET-INTEL-01`, hasn't been built yet), so nothing breaks. This is simply the shape any future caller must use.
2. **A `deliverables` row is now written on every call** (via `sendRequest()`'s `store` handler), which `channel-intelligence.js` never did. This is not a regression — it's `ARCHITECTURE.md` §2 Key Rule 7 ("every level can produce a Deliverable") finally applying to Marcus's calls, same as every other `sendRequest()`-routed capability already does. `task_id: null` (no Work Order involved) — the `deliverables` schema already allows this.
3. **Tool name/description sent to Anthropic becomes generic.** `sendRequest()`'s `buildCallBody()` always names the tool after `format_contract.skill_profile_slug` with a fixed description, "Return structured output" — not `channel-intelligence.js`'s specific `classify_intent`/`answer_question` names and descriptions. The enforced JSON schema (`properties`/`enum`/`required`) is unaffected — `tool_choice` still forces the exact schema regardless of name. This is the same convention every other `sendRequest()`-routed capability (`plan.js`) already uses; extending `request-receivable.js` to carry richer names would be a 4th file this session doesn't have budget for (Section 2 Session Scope Rules: max 3 files) — logged as a possible future polish item, not blocking.

### Task 3 — `api/capabilities/execute.js` (new) — the Generic Capability Executor

```js
// DeepBench v5.3.8 | api/capabilities/execute.js | S-CAPABILITY-EXEC-01 — the Generic Capability Executor (ARCHITECTURE.md §19b)
// FEATURE: AA-76 — one generic route for every AI-pattern capability. No capability-specific
// logic lives here, ever — model/max_tokens/schema come entirely from Skill Profile data via
// assemblePrompt() (AA-75). A new capability requires zero changes to this file — only new
// Supabase rows (Skill Profiles + capability_skill_profiles + agent_capability_assignments).

import { assemblePrompt } from '../prompt/db-assembly.js';
import { enrichPrompt } from '../prompt/ai-enrichment.js';
import { sendRequest } from '../prompt/request-receivable.js';

export const config = { maxDuration: 60, runtime: "nodejs" };

// FEATURE: AA-76 — core logic exported separately from the HTTP handler so the Node.js test
// can call it directly, same pattern as runChannelIntelligence/assemblePrompt/enrichPrompt.
export async function runCapability({
  capability_slug,
  intent_slug = null,
  agent_id,
  task_context,
  runtime_context = null,
  tenant_id = 'global',
  enrichment_capability_slug = null,
}) {
  if (!capability_slug) throw new Error('capability_slug required');
  if (!agent_id) throw new Error('agent_id required');
  if (!task_context) throw new Error('task_context required');

  const promptRequest = await assemblePrompt({
    capability_slug,
    agent_id,
    tenant_id,
    task_context,
    runtime_context,
    intent_slug,
    enrichment_capability_slug,
  });

  const enriched = await enrichPrompt({ prompt_request: promptRequest, agent_id, capability_slug });

  const result = await sendRequest({ prompt_request: enriched, agent_id, capability_slug, tenant_id });

  return result;
}

export default async function handler(req, res) {
  const allowedOrigin = process.env.ALLOWED_ORIGIN || "*";
  res.setHeader("Access-Control-Allow-Origin", allowedOrigin);
  res.setHeader("Access-Control-Allow-Methods", "POST, OPTIONS");
  res.setHeader("Access-Control-Allow-Headers", "Content-Type");
  if (req.method === "OPTIONS") return res.status(200).end();
  if (req.method !== "POST") return res.status(405).json({ error: "Method not allowed" });

  try {
    const result = await runCapability(req.body || {});
    return res.status(200).json(result);
  } catch (e) {
    console.error('[execute] error:', e);
    return res.status(e.status || 500).json({ error: e.message });
  }
}
```

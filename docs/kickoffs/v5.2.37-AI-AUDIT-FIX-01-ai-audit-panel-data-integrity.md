## Section 2: CONTEXT

Four data integrity and display correctness bugs were discovered in the AI Audit panel.

**BUG-20:** `prompt-service` appears as an LLM model in the "By LLM" section of the AI Audit panel with 16 calls. Root cause: 16 stale rows in `ai_activity_log` were written by pre-BUG-14 code (`logAICall({type:'planning', model:'prompt-service'})`) that was removed in S-CONTENT-01b. Those rows persist in Supabase. Fix is two-part: (1) delete the stale rows via Supabase MCP, (2) add a `KNOWN_LLM_MODELS` filter to the `byLLM` aggregation as a permanent defense against any future non-model strings reaching the By LLM section.

**BUG-20 also includes:** `MODEL_PROVIDER` is missing `claude-haiku-4-5-20251001` and `claude-sonnet-4-6` — the two model IDs actually in use. Both show "Unknown provider" in By LLM. An AI Transparency screen showing "Unknown provider" for Anthropic's own models is a credibility failure. Also: `logAICall()` call sites across the frontend use short-form model IDs (`claude-haiku-4-5`, `claude-sonnet-4-5`). A `MODEL_ID_NORMALIZE` map in `logAICall()` will canonicalize these to full versioned IDs at write time, keeping Supabase data clean going forward.

**BUG-21:** "By Agent" shows raw agent IDs (`dan`, `pp-01`) instead of formatted names. Root cause: `AGENT_NAMES` in `AIActivityPanel.jsx` is a static manually-maintained map that was never updated when Dan Bingham (PS-01) was added in S-DAN-02. The `pp-01` entry is a legacy DB row using the agent code instead of the agent `id` slug. Fix: replace `AGENT_NAMES` entirely with a lookup derived from the authoritative `AGENTS` array in `agents.js`. Build two maps at module level — one keyed by `id`, one by `code` — so both `dan` and `pp-01` resolve correctly. This map never goes stale because it is derived, not maintained.

**BUG-22:** The Platform Roadmap section lists four services that have already shipped as live: AI Enrichment, Request & Receivable, Prompt Assembly, Knowledge Reinforcement. The `roadmap` field in `SERVICE_CATALOG` was never updated when those services were built. Fix: update four `roadmap` values from `'next'` → `'now'`. A standing rule is added to `STANDARDS.md` Section 12 so this cannot recur.

```markdown
## Section 12: Canonical Model ID Standard

### Canonical Anthropic model IDs (always use these exact strings)

| Short-form (DO NOT USE) | Canonical (USE THIS) |
|-------------------------|----------------------|
| `claude-haiku-4-5` | `claude-haiku-4-5-20251001` |
| `claude-sonnet-4-5` | `claude-sonnet-4-6` |

**Rule:** Never use short-form model IDs in `logAICall()` call sites or server-side `ai_activity_log` inserts. Always use the canonical versioned string. `MODEL_ID_NORMALIZE` in `useAIActivity.js` normalizes legacy short-form IDs as a safety net — it is not a license to keep using short-form IDs in new code.

**Why:** The AI Audit panel groups by model string. Short-form and full-version IDs produce two separate rows for the same model, splitting cost and call counts. An AI Transparency screen with fragmented model data or "Unknown provider" labels fails its purpose.

### SERVICE_CATALOG roadmap field rule

When any `api/` route or service capability ships in a coding session, that session MUST update the corresponding `SERVICE_CATALOG` entry in `useAIActivity.js` from `roadmap: 'next'` (or `'later'`) to `roadmap: 'now'` in the same commit.

The Platform Roadmap section of the AI Audit panel shows all services where `roadmap !== 'now'`. A live service appearing in the roadmap is incorrect and will mislead any architect reviewing the panel.

**Add to CLAUDE-DESIGN.md AI Audit wiring checklist:** "SERVICE_CATALOG `roadmap` updated to `'now'` for any service that ships in this session."
```

## SECTION 2 — CONTEXT

This session seeds the Project Manager Capability into Supabase. All five tables required were created in S-SK-01. This session is **pure INSERT** — no CREATE TABLE, no schema changes, no src/ changes.

**What gets seeded:**
1. 5 Skill Profile rows in `skill_profiles` (SP-PM-01 through SP-PM-05)
2. 1 Capability row in `capabilities` (CAP-PM-01 `project-manager`)
3. 5 join rows in `capability_skill_profiles` (one per Skill Profile, with Level)
4. 1 agent assignment row in `agent_capability_assignments` (Michelle PP-01 → `project-manager`)

**What this unlocks:** Michelle Manning's Personnel File Profile tab immediately shows the Project Manager Capability and all 5 Skill Profiles via the existing SK-06 UI (SkillHoverCard, level badges, type chips). No code changes required.

**What this does NOT include:**
- No `deliverables` table (S-DELIVER-04 / DL-04)
- No `capability_gaps` table (S-PM-02 or S-DELIVER-04)
- No `prompt-assembly.js` (S-PM-02 / AA-03)
- No RAG embedding of capability records into `knowledge_entries` — SP-PM-05 (Capability Registry Knowledge) declares the intent; embedding is wired in the Work Side execution sprint
- No `technical_services` wired — all 5 Skill Profiles seed with `technical_services: []`

**Michelle transition context:** This session makes CAP-PM-01 visible on Michelle's Personnel File. It does NOT remove or replace the hardcoded planning logic in `api/plan.js` — that happens in S-PM-03 after `prompt-assembly.js` is built.

## 2. CONTEXT

S-LIBRARIAN-01a shipped `lib/librarian.js`'s `queryLibrary()` broker and Eleanor Voss's persona (commit `542b1fe`, QA passed) — but nothing calls her yet. This session wires her into the one real call site (`api/prompt/ai-enrichment.js`'s `fetchSection()`) via the opt-in `fetch_instruction.broker` field locked in the 01a design, and fixes an AI Audit attribution gap discovered in this session's Architect Review.

**The bug found in already-shipped code:** `lib/librarian.js`'s `logLibrarianCall()` writes `ai_type: "deterministic"` on every Eleanor call. `AI_TYPE_TO_SERVICE` (`useAIActivity.js`) has no entry for the literal string `"deterministic"`, and no `SERVICE_CATALOG` slug is named `"deterministic"` either — so `byService`'s lookup (`AI_TYPE_TO_SERVICE[e.type] || e.type`) falls through to `"deterministic"` itself, which matches no catalog entry. Simply adding a `librarian` `SERVICE_CATALOG` entry would not fix this — Eleanor's calls would still never attribute to it. The fix follows the exact precedent every other named service already uses (`reflect`, `synthesis`, `ai_enrichment`, `db_assembly`, `request_receivable`): a distinct `ai_type` string per service, mapped through `AI_TYPE_TO_SERVICE`. Confirmed via direct file read (not assumed): no other file in the codebase currently writes `ai_type: "deterministic"` from an actual call site (`ARCHITECTURE.md` §1 documents it as the convention for deterministic capabilities, but no code — not `db-assembly.js`, not `procurement-flags`, not this file until now — actually emits that literal value from a live call site). This is a one-off gap fully contained in `lib/librarian.js`, not a platform-wide pattern to reconcile.

**The wiring (locked in 01a's design, unchanged here):** `fetchSection()`'s `"rag"` branch gets one new check: if `section.fetch_instruction.broker === "librarian"`, call `queryLibrary({ requestingAgentId, queryText: taskContext, tenantId, matchCount: fi.match_count || 5 })` instead of `queryRAG(...)`. `requestingAgentId` is `enrichPrompt()`'s existing `effectiveAgentId` — the trusted server-side agent identity already computed at line 83, never LLM output or client input (this is what makes 01a's credential check real). Every existing caller's `fetch_instruction` has no `broker` field, so `fi.broker === "librarian"` is `false` for all of them — the branch falls through to the exact `queryRAG(...)` call that runs today, byte-identical. No Skill Profile in Supabase sets `broker: "librarian"` yet (that's S-APPLE-02/03's job), so this session changes zero production behavior for any live agent — it only makes the mechanism callable.

**Not in scope this session** (deferred to S-LIBRARIAN-01c, immediately next):

- Any new "Eleanor collaborated" UI indicator — the AI Audit panel (via her own logged row, fixed this session) is the only visibility surface for now; a UI collaboration chip is not part of either 01b or 01c's named scope

Change `fetchSection()`'s signature and its `"rag"` branch:
```js
async function fetchSection(section, taskContext, tenantId, requestingAgentId) {
  if (section.type === "stored") return { ...section };

  if (section.type === "rag") {
    const fi = section.fetch_instruction;
    try {
      // FEATURE: AG-27 — broker opt-in: fi.broker === "librarian" routes through Eleanor's
      // credential-checked queryLibrary() instead of a direct queryRAG() call. Default
      // (fi.broker unset) is byte-identical to today's behavior for every existing caller.
      const result = fi.broker === "librarian"
        ? await fetchWithTimeout(
            queryLibrary({
              requestingAgentId,
              queryText: taskContext,
              tenantId,
              matchCount: fi.match_count || 5,
            }),
            RAG_TIMEOUT_MS
          )
        : await fetchWithTimeout(
            queryRAG({
              queryText: taskContext,
              agentId: fi.agent_id || null,
              tenantId,
              matchCount: fi.match_count || 5,
              scope: fi.scope || "agent",
            }),
            RAG_TIMEOUT_MS
          );
      return {
        ...section,
        content: result.context || "",
        _rag_chunks: result.matchCount || 0,
        _rag_scope_effective: (fi.scope === "agent" && fi.agent_id) ? "agent" : "platform",
        _librarian_tier: result._librarian?.tier || null,
      };
    } catch (e) {
      console.warn(`[ai-enrichment] RAG fetch failed for section ${section.slug}:`, e.message);
      return { ...section, content: "", _fetch_error: e.message };
    }
  }

  // reflect and synthesis sections: skip in Step 1, handled in Steps 3+4
  return { ...section };
}
```

Update the call site (inside `enrichPrompt()`, currently `nonReflectSections.map(s => fetchSection(s, taskContextStr, tenant_id))`) to pass the fourth argument:

  const fetchedSections = await Promise.all(

No other line in this file changes. `_librarian_tier` is an additive, optional debug field (mirrors the existing `_rag_scope_effective` pattern) — `null` for every non-broker call, harmless if unused.

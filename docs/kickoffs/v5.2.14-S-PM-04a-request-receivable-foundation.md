## SECTION 2 — CONTEXT

S-PM-04a is the foundation session before the Request & Receivable service is built in S-PM-04b.

It does four things:
1. Creates the `deliverables` table and adds `patterns_used` column to `ai_activity_log` in Supabase
2. Updates the `execution-plan` skill profile in Supabase (add `handler`, `title`, `taskTitle` to traits)
3. Merges `title.js` into `plan.js` (frees one Vercel slot for S-PM-04b's new route) and adds `taskTitle` to the plan tool schema
4. Patches `db-assembly.js` + `ai-enrichment.js` + `useAIActivity.js` — small targeted changes so S-PM-04b can build against correct interfaces

**What this does NOT include:**
- No `request-receivable.js` — that is S-PM-04b
- No `api/lib/handlers/store.js` — that is S-PM-04b
- No frontend wiring to the Prompt Service pipeline — that is S-PM-05 (design session first)

**Why title.js is merged now:**
`title.js` is a temporary hardcoded shortcut for functionality that will move into the Prompt Service pipeline (S-PM-05). Merging it into `plan.js` via an `action` param frees one Vercel slot so S-PM-04b can add `request-receivable.js` without exceeding the 12-function Hobby limit. When S-PM-05 wires the frontend to the Prompt Service, `plan.js` is deleted entirely.

**Why `taskTitle` is added to the plan tool schema:**
`title.js` currently generates `taskTitle` separately. When merged, the title action continues to return `{ taskTitle, stepTitles }` unchanged. Additionally, the execution-plan Format Skill schema gains a `title` field (deliverable title, generated by the LLM from response content) and `taskTitle` in `output_fields`. These are distinct: `taskTitle` = work order title (max 8 words, action-oriented), `title` = deliverable title (max 8 words, describes what was produced).

### Task 1 — Supabase: deliverables table + ai_activity_log patch + execution-plan UPDATE

Run as three separate SQL blocks via Supabase MCP `execute_sql`.

**Block 1 — Create deliverables table:**
```sql
create table if not exists deliverables (
  id                  uuid primary key default gen_random_uuid(),
  tenant_id           text not null,
  task_id             uuid,
  step_id             uuid,
  agent_id            text,
  skill_profile_slug  text,
  type                text,
  title               text,
  content             jsonb,
  format              text,
  status              text default 'draft',
  handler             text default 'store',
  level               int,
  is_final            boolean default false,
  version_of          uuid,
  consumes            jsonb,
  is_public           boolean default false,
  is_shared           boolean default false,
  share_token         text,
  price_usd           numeric,
  created_at          timestamptz default now()
);
```

**Block 2 — Add patterns_used to ai_activity_log:**
```sql
alter table ai_activity_log
  add column if not exists patterns_used jsonb;
```

**Block 3 — UPDATE execution-plan skill profile traits:**
```sql
update skill_profiles
set traits = traits
  || '{"handler": "store"}'::jsonb
  || jsonb_build_object(
       'output_fields',
       (coalesce(traits->'output_fields', '[]'::jsonb))
         || '["title", "taskTitle"]'::jsonb
     )
where slug = 'execution-plan';
```

**Verify after each block:** confirm no errors. Run `select id, traits from skill_profiles where slug = 'execution-plan'` to confirm traits updated correctly.
